# Security Audit Report - 2026-07-05

## Scope

Reviewed the application security posture across authentication, authorization, session handling, CSRF, XSS, SQL injection, file upload, permission bypass, API authorization, password handling, and input validation.

This pass used local static analysis and automated local verification. Production/staging database credentials were not used in this turn, so database-backed authenticated E2E abuse cases were not executed here.

## Findings Fixed

| Area | Finding | Fix |
|---|---|---|
| App secret / CSRF | Source contained a hardcoded fallback application key, making CSRF HMAC predictable if `APP_KEY` was missing. | `config/app.php` now uses `APP_KEY` when configured; otherwise creates and persists a runtime secret in `uploads/.app_key`, failing closed if it cannot persist. |
| Security headers | Security headers depended mainly on Apache config. | Added PHP-level security headers including CSP, frame ancestor restriction, nosniff, referrer policy, and permissions policy. |
| File download | Attachment download trusted `file_path` from DB without realpath confinement. | `FileController` now resolves paths and only serves files under `uploads/`. |
| File upload | Generic file upload did not validate upload error/empty file/extension before MIME mapping. | Added upload error, size, extension, MIME, SVG size, and XML non-network parsing guards. |
| Media upload/SVG | UI media SVG validation was limited and SVG responses had no dedicated CSP. | Added upload validation, SVG size/content/XML checks, realpath prefix hardening, nosniff, and SVG-specific CSP. |
| Backup restore | Restore accepted arbitrary `.sql` payloads for privileged users. | Restore now requires SQL generated by this application and blocks privileged MySQL statements such as user/grant/database/outfile operations. |
| Import upload | CSV/XLSX parsing lacked upload error checks, MIME validation, XML non-network parsing, and row/XML expansion limits. | Added upload/MIME checks, row cap, XML size caps, and `LIBXML_NONET` parsing. |
| Password handling | Password policy did not cap maximum length and did not rehash old hashes. | Added 8..1024 byte policy and automatic `password_needs_rehash()` update after successful login. |
| File listing API | `GET /api/files` could hit a method argument mismatch and produce an uncontrolled server error. | Made file listing validate `module` and `entityId` from query parameters and return controlled 422 on invalid input. |
| Rich text XSS | Frontend rich text sanitizer removed scripts/events but allowed more risky attributes/schemes. | Hardened sanitizer to remove additional active elements/attributes and allow only safe URL schemes. |

## Controls Confirmed

- Authentication uses `password_hash()` / `password_verify()` and random 32-byte session tokens.
- Session tokens are stored as SHA-256 hashes and checked for expiry/revocation.
- Mutating authorized endpoints call `verifyCsrfToken()` through `requirePermission()` or `requireAdmin()`.
- PDO uses prepared statements and `ATTR_EMULATE_PREPARES=false`.
- Upload directories are blocked from direct web access by `.htaccess`; downloads/media are routed through validated endpoints.
- Controllers for private modules consistently call `requirePermission()` or `requireAdmin()`.

## Remaining Risk / Follow-up

- A full authenticated penetration test against production/staging data was not executed in this turn.
- Backup restore remains an intentionally privileged destructive operation. It is now constrained to application-generated SQL, but should remain Super Admin only operationally.
- CSP currently permits inline script/style because the existing UI uses inline handlers and inline scripts. A future hardening pass should remove inline handlers and move to nonce/hash based CSP.
